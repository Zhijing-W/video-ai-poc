# 人脸超分对比实验设计

> 当前状态：先记录产品质量结果到后续动作的映射。实验数据、对比方案、指标和命令后续继续补充。

## 1. 设计原则

人脸检测后只运行一次产品质量评估。质量评估输出总体等级和具体问题标签，超分模块直接消费这些
结果，不再建立另一套重复的人脸质量评分逻辑。

## 2. 质量结果与处理动作

| 质量结果 | 建档 | 查询匹配 | 超分 |
|---|---|---|---|
| `clear` | 是 | 是 | 否 |
| `marginal` | 否或谨慎 | 是，适当降权 | 有模糊标签时尝试 |
| `poor + blur/small_face` | 否 | 可以低权重查询 | 可以尝试 |
| `poor + extreme_pose` | 否 | 谨慎或拒绝 | 否 |
| `none` | 否 | 人脸路线不匹配 | 当前人脸裁剪超分无法运行 |

`none`只表示人脸路线不可用，不会丢弃人物轨迹。系统仍继续运行人物跟踪、人形ReID、步态识别
和事件理解；如果无法确认身份，事件报告可以输出“未识别人物正在执行某行为”。

## 3. 建议的代码输出

```json
{
  "category": "poor",
  "defects": ["blur", "small_face"],
  "can_enroll": false,
  "can_match": true,
  "can_superres": true
}
```

总体等级使用：

```text
clear / marginal / poor / none
```

具体问题允许同时存在，例如：

```text
blur / small_face / extreme_yaw / extreme_pitch / low_detection / occlusion
```

## 4. FIQA接入位置

### 4.1 当前代码状态

当前项目已经预留FIQA配置和函数接口，但尚未加载任何真实FIQA模型：

```text
app/core/config.py
└─ FACE_FIQA_BACKEND

app/face.py
├─ _deep_fiqa_score(...)
└─ assess_quality(...)
```

`_deep_fiqa_score()`目前是占位实现，实际返回`None`。当前质量判断主要使用人脸尺寸、拉普拉斯
方差、人脸检测分数和5点关键点估算的yaw/pitch。

### 4.2 建议的模型加载位置

在`app/face.py`中增加独立的FIQA模型状态和懒加载函数，位置可与`_ensure_superres()`并列：

```python
_fiqa_state = {
    "backend": None,
    "model": None,
    "error": None
}


def _ensure_fiqa():
    """根据FACE_FIQA_BACKEND加载CR-FIQA、SER-FIQ或OFIQ。"""
```

模型应在进程中只加载一次，不能每处理一帧重新加载。

### 4.3 推理调用位置

推荐处理顺序：

```text
SCRFD检测到脸
→ 5点关键点对齐
→ FIQA对对齐人脸输出身份识别可用性分数
→ 与尺寸、拉普拉斯、姿态、检测分数一起进入assess_quality()
→ 输出category、defects和动作字段
```

质量输出建议保留：

```json
{
  "category": "poor",
  "quality_score": 0.31,
  "fiqa_score": 0.24,
  "defects": ["blur", "small_face"],
  "can_enroll": false,
  "can_match": true,
  "can_superres": true
}
```

### 4.4 超分后的FIQA复评

当`can_superres=true`时：

```text
原始对齐脸
→ GFPGAN或CodeFormer
→ 使用同一个已加载FIQA模型重新评分
→ 比较修复前后FIQA分数
→ 检查ArcFace身份特征是否发生漂移
→ 质量提高且身份稳定时，才使用修复后特征
```

超分后复评只重新运行FIQA推理，不重新加载模型。
