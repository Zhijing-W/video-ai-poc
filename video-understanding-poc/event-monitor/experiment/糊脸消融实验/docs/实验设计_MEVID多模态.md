# MEVID 身份实验设计 v5：受控模型实验、产品状态机回放与全帧验证

## 0. 正式结果与统一命名

当前论文正式使用：

- **协议A**：比较人脸、人形和步态分别能提供多少身份信息；
- **协议B3**：使用训练人员确定固定身份匹配判断标准，再在完全不同的测试人员上比较人脸方案。

协议B和B2只保留为设计演进记录，不作为最终论文结论。

为避免同一术语被不同读者理解成不同含义，本文统一使用以下名称：

| 不单独使用的简称 | 正式写法 |
|---|---|
| 训练目标1% | **严格档：训练人员校准数据中的陌生人误识率上限1%** |
| 训练目标5% | **平衡档：训练人员校准数据中的陌生人误识率上限5%** |
| 训练目标10% | **宽松档：训练人员校准数据中的陌生人误识率上限10%** |
| TPIR | **库内人员正确识别并接受率（TPIR）** |
| FMR | **陌生人误识率（FMR）** |
| Rank-1 | **强制选择第一名时的身份正确率（Rank-1）** |
| gallery | **参考候选片段或参考库** |
| query/probe | **待识别监控片段** |
| anchor | **参考照片** |
| template | **数字参考特征** |
| genuine | **库内同人样本** |
| imposter | **库外陌生人样本** |
| canonical subject | **统一主体ID** |

其中1%、5%和10%限制的是校准数据中陌生人被错误接受的比例，不是模型训练准确率，也不是
测试集最终保证值。产品最终只选择一种档位对应的固定相似度阈值。

> 上一版把“严格产品建档门控”和“ArcFace/AdaFace/超分模型对比”放在同一次运行中。
> MEVID 监控人物 crop 中清晰脸极少，导致 27 个建档身份只有 1 个能建立人脸模板；
> 人脸模型几乎没有参与最终判定，因此 M0–M3 完全相同。v4 将问题拆成两个独立协议。

> 本协议复用产品模型和核心质量门控，但不是“原始视频输入到最终事件报告”的完整产品端到端运行。
> MEVID test 提供的是已经裁好并按轨迹组织的人物图片，因此检测、跟踪、实时建库顺序和
> canonical subject registry 等产品状态逻辑无法原样复现。


### 2.0 与产品相同和不同的部分

**直接复用产品实现：**

- 人脸检测、ArcFace embedding 与 `assess_quality` 质量分桶；
- 人脸 `clear-only` 建档门控；
- OSNet 人形 embedding、`reid.assess_quality` 和 `gallery.quality_ok`；
- SkeletonGaitPP 步态提取及最少有效帧要求；
- 人脸质量低时降权、人形和步态兜底的总体思路；
- fresh run、开放集冒充者、禁止 query 进入 gallery。

**为离线评测而改变、尚未与产品完全相同：**

- 输入是 MEVID 已裁好的人物轨迹，不运行原始全帧 YOLO 检测、ByteTrack 跟踪和人脸-人体关联；
- 使用 MEVID 真值 `person_id` 将三路模板绑定到同一 canonical identity；
- 每个身份对多条 gallery 轨迹求均值模板，不是产品按时间顺序执行
  `identify_or_enroll`、hit/grey/new、多 shot 淘汰和负缓存；
- 实验一次均匀抽取固定帧数，不是产品按轨迹最佳帧和三时钟实时触发；
- 融合使用 canonical PID 上的独立 top-1 投票，产品侧尚缺正式跨模态 subject registry；
- 实验扫描完整 DET 曲线，产品运行时使用预先校准并冻结的固定阈值；
- 不运行 LLM 事件理解，因为本实验只评估身份识别。

因此协议 A 应理解为“产品核心模型与门控对齐的身份离线实验”，不能表述为与当前 Demo
逐行、逐状态完全相同。

### 2.1 研究问题

在不放松产品质量门控的前提下：

1. 严格人脸门控会损失多少建档覆盖率？
2. 人形和步态分别能救回多少无脸、糊脸身份？
3. 加入更多模态后，开放集误识是否上升？
4. 步态在换装场景中是增益还是噪声？

本协议不用于比较 ArcFace 与 AdaFace。人脸模型固定为 ArcFace，避免“脸根本没有入库”
却错误解释为“不同人脸模型效果相同”。

### 2.2 Arms

| arm | 人脸 | 人形 | 步态 | 目的 |
|---|---:|---:|---:|---|
| F | ✓ |  |  | 严格人脸门控的真实下限 |
| B |  | ✓ |  | OSNet 单路能力 |

### 2.3 建档

- 人脸：只使用官方 gallery 轨迹中 `category == clear` 的帧。
- 人形：使用 `app.reid.assess_quality` + `app.gallery.quality_ok` 通过的帧。
- 步态：只有成功产生 SkeletonGaitPP 向量的轨迹才入库。
- 每个身份、每个模态分别对多条 gallery 轨迹求均值模板。
- 必须报告 `face/body/gait` 三路各自建档覆盖率。

### 2.4 正确融合

#### 先统一身份空间

人脸、人形、步态 gallery 的本地自增 ID **不是同一套编号**，不能直接比较：

```text
face_gallery_id=3  ≠  body_gallery_id=3  ≠  gait_gallery_id=3
```

本实验是有 MEVID 身份真值的离线评测，因此在建库时直接使用同一套官方 `person_id`
作为三路模板的 canonical key。例如身份 `0201`：

```text
face_templates["0201"] = 该人的人脸模板
body_templates["0201"] = 该人的人形模板
gait_templates["0201"] = 该人的步态模板
```

所以实验中每一路 top-1 输出的是**已映射后的 MEVID canonical person_id**：

```text
face  -> (canonical_pid="0201", face_score)
body  -> (canonical_pid="0201", body_score)
gait  -> (canonical_pid="0298", gait_score)
```

此时 face 与 body 都预测 canonical `0201`，可以算两路一致；gait 预测 `0298`，不能参与
`0201` 的一致奖励。

#### 产品运行时如何做到

真实 Demo 没有 MEVID 标签，必须增加跨模态 subject registry，在同一条高可信 track
同时产生多种模板时建立绑定：

```text
canonical_subject_id=42
  ├─ face_gallery_id=3
  ├─ body_gallery_id=8
  └─ gait_gallery_id=5
```

查询时先分别查各自 gallery，再通过 registry 映射：

```text
face local#3 -> canonical#42
body local#8 -> canonical#42
gait local#2 -> canonical#17
```

只有映射后 canonical ID 相同的路线才能增加一致奖励。若某个本地 ID 尚未绑定到
canonical subject，则该路线可以提供局部相似度，但不能声称与另一模态“身份一致”。

不同模态若预测不同身份，则各自为对应身份投票；未预测该身份的路线不为它贡献分数。

### 2.5 指标

#### 建档覆盖率：face/body/gait

分母是本次计划建档的身份数，分子是该模态至少成功建立一个合格模板的身份数：

```text
face coverage = 成功建立人脸模板的身份数 / 建档身份总数
body coverage = 成功建立人形模板的身份数 / 建档身份总数
gait coverage = 成功建立步态模板的身份数 / 建档身份总数
```

例如 `face/body/gait = 1/27/23` 表示：27 个建档身份中，只有 1 人出现过满足
`clear-only` 门控的人脸，27 人都有人形模板，23 人有足够有效序列建立步态模板。

覆盖率只回答“能否进入该模态的库”，不回答“查询时能否认对”。覆盖率低可能来自无脸、
太糊、人体 crop 质量不合格或步态有效帧不足。

#### 强制选择第一名时的身份正确率（Rank-1）

只对库内 genuine probe 计算，并且强制从所有建档身份中选相似度最高的一个：

```text
rank-1 = top-1 身份正确的 genuine probe 数 / genuine probe 总数
```

它不允许系统回答“不认识”，也不包含库外冒充者，因此只用于观察 embedding 的身份区分能力。
rank-1 高不代表开放集系统安全：即使所有陌生人都会被强行认成库内某人，闭集 rank-1 也不受影响。

#### 陌生人误识率（FMR）

```text
FMR = 被错误接受为任一库内身份的 imposter probe 数 / imposter probe 总数
```

越低越好。MEVID 当前约有 87 条 imposter query 轨迹时，一个误接受约等于
`1/87 = 1.15%` FMR。因此：

- 陌生人误识率上限1%：实际要求0条误接受；
- 陌生人误识率上限5%：最多允许4条误接受（4/87≈4.6%）；
- 陌生人误识率上限10%：最多允许8条误接受（8/87≈9.2%）。

#### 库内人员正确识别并接受率（TPIR）

```text
TPIR = 身份 top-1 正确且分数达到接受阈值的 genuine probe 数 / genuine probe 总数
```

TPIR 同时要求“认对人”和“置信度过阈值”。如果 top-1 正确但分数太低而被拒识，也不计入 TPIR。

#### 在不同陌生人误识率上限下的库内人员正确识别并接受率

不断提高或降低接受阈值会形成“多认出一些人”和“多误接受陌生人”的权衡。对每个目标 FMR，
在满足误接受上限的阈值中报告最高 TPIR：

```text
平衡档结果 = 在陌生人误识率不超过5%时，库内人员正确识别并接受的比例
```

这是有陌生人场景下的主指标，越高越好。不同方案的原始相似度分布不同，因此各自通过完整曲线寻找
对应 operating point，不能统一套用未经校准的 `0.5`。

实验曲线用于比较模型；上线使用的固定阈值仍应在独立校准集上确定，不能直接拿测试集最佳阈值部署。

#### clear/marginal/poor/none 各质量桶 TPIR

质量桶只根据 **query 原始监控帧的人脸质量**划分：

- `clear`：脸尺寸、角度和清晰度均较好；
- `marginal`：可用但不够理想；
- `poor`：过小、模糊或角度极端；
- `none`：采样帧中没有检测到脸。

在同一个全局 operating threshold 下，分别计算各桶 TPIR，而不是每个桶单独挑阈值。
`none` 桶中纯人脸 arm 理论上接近 0；人形/步态能否提高该桶 TPIR，是多模态兜底价值的直接证据。

#### DET 曲线

遍历全部接受阈值，记录：

```text
x = FMR
y = FNIR = 1 - TPIR
```

曲线越靠左下越好。JSON 同时保存 `FMR / TPIR / FNIR / threshold`，既可画标准 DET，
也可画更直观的 TPIR-FMR 权衡曲线。单个固定阈值只是一点，完整曲线可以防止因阈值选择不公平
而误判模型优劣。

#### 逐 probe 路线预测

每条 query 轨迹保存：

```text
gt / genuine-or-imposter / quality_bin
face、body、gait 各自的 top-1 canonical PID 与分数
最终 fused PID、置信度、实际达成一致的路线
```

它不作为一个汇总指标，而是用于定位 bad case：判断错误来自人脸未检出、人形换装混淆、
步态序列失效、某一路高分认错，还是融合阶段选择错误。

`IDENTITY_RESOLVE_THRESH=0.5` 不再作为实验固定结论阈值。实验通过 DET 扫描选择目标
FMR 下的 operating point，避免把未经校准的产品默认值直接套到新数据集和新模型上。

## 3. 协议 B：固定锚点的人脸模型实验

入口：


### 3.1 研究问题

在每个身份使用完全相同的清晰登记照输入时，公平比较：

| variant | 人脸后端 | 超分 |
|---|---|---:|
| FR0 | ArcFace |  |
| FR1 | AdaFace |  |
| FR2 | ArcFace | GFPGAN |
| FR3 | AdaFace | GFPGAN |

人形和步态不参与本协议，防止它们掩盖人脸模型差异。

### 3.2 固定官方 gallery 锚点

> **这不是产品冷启动/自动积累入库流程。** 协议 B 是受控的人脸模型 benchmark：
> 先为每个已知身份准备一组固定参考图片，再让 ArcFace、AdaFace、GFPGAN 使用完全相同的
> 参考输入和 query，避免建档门控及人形/步态掩盖人脸模型差异。
>
> 这些参考图片也不是 `actor_checkin` 提供的登记正脸照，而是从 MEVID 官方 gallery
> 监控人物轨迹中挑出的“该身份当前可找到的最佳人脸帧”。它们仍可能是远场、侧脸或低清，
> 只是相对于该身份其他 gallery 帧质量更高。

- 官方公开资料没有提供 `actor_checkin` 文件名前缀到 test `person_id` 的显式映射表，
  因此正式实验不使用未经证明的直接补零映射。
- 锚点只从官方 gallery 轨迹中选择，任何 `query_IDX.txt` 指定的 query 轨迹都禁止进入锚点候选。
- 每个身份默认取前 3 条 gallery 轨迹，每条均匀抽取 8 个候选帧。
- 使用原始 ArcFace 检测与产品质量分对候选帧排序，确定性选择质量最高的 3 张人脸锚点。
- 锚点选择只使用检测和质量，不使用 query，也不根据某个识别模型的匹配结果挑图。
- 所有 variant 使用完全相同的原始锚点图片和 query 帧。
- 各模型在自己的 embedding 空间分别建模板，不能跨模型混用向量。
- 该协议不走 `clear-only` 自动建档门控；检测成功后按质量加权聚合。
- 脚本必须报告每个 variant 的固定 gallery 锚点模板覆盖率。

因此两条协议的建档含义不同：

```text
协议 A：按视频轨迹和产品质量门控逐步建档，回答“真实系统能不能入库、能否被多模态救回”。
协议 B：离线固定参考模板，回答“排除建档差异后，哪个人脸识别/超分方案更强”。
```

### 3.3 超分协议

- query 质量桶永远根据原始监控帧计算，保证 FR0–FR3 使用相同 bad-case 分桶。
- GFPGAN 只处理原图中被判定为模糊且角度不过于极端的对齐脸。
- 超分后的对齐脸分别进入 ArcFace 或 AdaFace。
- AdaFace+GFPGAN 不得再被原图 AdaFace embedding 覆盖。
- 输出实际检测帧数和实际增强帧数，确认超分 arm 不是空跑。

### 3.4 指标

- 固定参考照片形成数字参考特征的身份覆盖率。
- 各质量组中，强制选择第一名时的身份正确率。
- 严格档、平衡档和宽松档下的库内人员正确识别并接受率与陌生人实际误识率。
- DET 曲线。
- 实际检测帧数、超分触发帧数。

## 4. 下一轮改进实验

### 4.1 是否继续使用 MEVID

继续使用 MEVID。它仍适合研究：

- 真实监控人物轨迹中的小脸、模糊脸、侧脸和无人脸问题；
- 跨摄像头、跨日期和换装条件下的人形 ReID；
- 连续轨迹上的步态特征；
- gallery/query 隔离和库外冒充者开放集评测。

但 MEVID test 包是已经裁好的人物 crop，不能验证全帧人物漏检、ByteTrack 身份切换和
人脸-人体关联。因此后续分为三层：B2 做公平模型比较，C 做产品身份状态机回放，最后再使用
原始全帧视频完成端到端验证。

### 4.2 协议 B2：改进的人脸模型对比

目标仍是比较 ArcFace、AdaFace 和人脸修复，不模拟产品自动建档。

#### 数据和模板

1. 使用同一个人脸检测器，在实验开始前确定“能够建立共同参考模板”的身份集合；
2. 所有模型使用完全相同的官方 gallery 图片和 query 帧；
3. 锚点仅依据检测、尺寸、模糊和姿态质量选择，不根据任何识别模型是否认对来选图；
4. 对无法建立共同锚点的身份单独报告覆盖失败，不将其混入模型识别准确率；
5. 若后续获得官方确认的 `actor_checkin` 映射，则增加“高清登记照锚点”作为第二个条件，
   与“监控 gallery 锚点”并列报告，不能替换或混合。

#### 阈值校准与重复实验

1. 优先下载 MEVID `bbox_train`，在 train 身份上构造独立的锚点、genuine 和 imposter，
   为 ArcFace、AdaFace 分别校准阈值；
2. 阈值确定后冻结，再在官方 test 上计算最终 TPIR/FMR；
3. 若暂不下载 train，则在 test 身份中进行嵌套划分：校准身份和最终评测身份严格分离；
4. 至少运行 5 个身份划分种子，报告均值、标准差和 95% 置信区间，避免当前少数轨迹造成偶然结论。

#### 质量分桶

除 `clear/marginal/poor/none` 外，新增按主要退化原因分桶：

- `small-face`：脸尺寸不足；
- `blur`：检测到脸但拉普拉斯/FIQA 清晰度低；
- `pose`：yaw 或 pitch 较大；
- `none`：未检测到脸。

这样可以判断 AdaFace 的变化来自哪类低质量样本，而不是把尺寸、模糊和姿态全部混在 `poor` 中。

#### GFPGAN 专项

1. 仅使用 MEVID 中真实、已检测到的模糊脸，不人工合成模糊；
2. 预先统计符合超分条件的真实样本数，样本不足时不报告总体增益结论；
3. ArcFace/AdaFace 的原图与修复图使用同一锚点和同一 query；
4. 同时报告触发率、第一名身份正确率、不同陌生人误识率上限下的库内人员正确识别并接受率、
   修复前后相似度变化和身份漂移率；
5. 若修复提高视觉质量但增加陌生人误识率，则判定不适合进入身份识别主链路。

#### B2 已实现与最终结果（2026-07-13）

实现入口：

```text
run_mevid_face_b2.py
```

本次由于尚未下载 `bbox_train`，采用 test 内**轨迹级隔离**：

- 50 个身份同时具有 gallery 和至少 2 条 query；
- 46 个身份在 FR0–FR3 中都能使用同一组原始 gallery 图片建立模板；
- 运行 5 次身份重划分；
- 每次 27 个建档身份、23 个库外身份；

- 每次评测集：27 条 genuine query + 23 条 imposter query；

- 95% CI 使用小样本 t 分布计算。

运行命令：

```bash
  --calibration-query-tracks 1 \
```

| 方案 | 第一名身份正确率均值（95%置信区间） | 严格档：库内人员正确识别并接受率 | 平衡档 | 宽松档 |
|---|---:|---:|---:|---:|
| FR0 ArcFace | **21.5%**（17.6%–25.3%） | 0.7% | **5.2%** | **5.9%** |
| FR1 AdaFace | 20.7%（18.2%–23.3%） | **1.5%** | 2.2% | 3.0% |
| FR2 ArcFace+GFPGAN | **21.5%**（17.6%–25.3%） | 0.7% | **5.2%** | **5.9%** |
| FR3 AdaFace+GFPGAN | 20.7%（18.2%–23.3%） | **1.5%** | 2.2% | 3.0% |

在平衡档（校准陌生人误识率上限5%）下：

- ArcFace的评测陌生人实际误识率均值为3.5%，库内人员正确识别并接受率为5.2%；
- AdaFace的评测陌生人实际误识率均值为1.7%，库内人员正确识别并接受率为2.2%；
- 两者 Rank-1 的置信区间明显重叠，不能声称存在统计显著的闭集差异；
- ArcFace 在当前校准方法下取得更高开放集 TPIR，但方差仍较大。

GFPGAN在特征提取阶段只增强极少量独立帧；这些帧在5次身份划分中会被重复计入评测记录，未救回任何Rank-1
错误，也未造成 Rank-1 退化。FR2/FR3 与各自 baseline 的汇总结果完全相同，因此仍不能评价
GFPGAN 的总体有效性。

退化标签允许重叠。5 次评测共 135 个 genuine 轨迹实例，其中 48 个实例未检测到脸，
50 个带小脸标签、50 个带姿态偏转标签，仅极少数最佳脸满足独立 blur 标签。结果进一步说明
MEVID 当前人脸问题主要由**脸过小、姿态和完全未检出**共同造成，而不仅是可通过超分处理的模糊。

> 限制：阈值虽然没有使用评测轨迹，但校准与评测仍来自同一批测试身份。不同陌生人误识率上限
> 下的库内人员正确识别并接受率，不能与使用独立训练身份校准的文献结果直接比较。

### 4.3 协议 B3：独立训练身份校准（当前最终版）

训练人物裁剪包已下载并解压：

- 压缩包：32.68GB；
- 解压后：39GB；
- 104 个训练身份、6338 条有效轨迹；
- 3,609,156 张图片，与 `train_name.txt` 行数完全一致；
- 训练身份与测试身份无交集。

入口：

```text
run_mevid_face_b3.py
```

#### 校准与评测

训练集：

- 50 个身份建立共同参考人脸库；
- 其余 54 个身份全部作为数据库外人员；
- 每个库内身份最多使用 8 条独立轨迹提供同人校准样本；
- 每个库外身份最多使用 8 条轨迹提供陌生人校准样本；
- 共提取 400 条同人轨迹和 410 条陌生人轨迹；
- 其中 223 条同人、209 条陌生人轨迹成功产生可评分人脸特征；
- 每个模型据此确定一套全局阈值，随后冻结。

测试集：

- 52 个具有官方 gallery/query 的身份；
- 48 个身份在 FR0–FR3 中都能建立共同参考模板；
- 每次 27 个身份进入参考库，其余 25 个作为数据库外人员；
- 官方 query 只计算最终成绩，不再调整阈值；
- 重复 5 次身份划分。

人脸模型主指标只统计成功产生人脸特征的轨迹，避免完全未检测到脸的样本人为降低FMR；
同时另行报告全部轨迹的人脸查询覆盖率。5次评测中，库内轨迹的人脸可评分覆盖率约64%，
数据库外轨迹约64%。

#### 用训练人员校准后固定的身份匹配阈值

| 模型 | 严格档：校准陌生人误识率≤1% | 平衡档：≤5% | 宽松档：≤10% |
|---|---:|---:|---:|
| ArcFace | 0.5916 | 0.5494 | 0.5045 |
| AdaFace | 0.6433 | 0.5647 | 0.5174 |

#### 测试集结果

三种档位限制的是训练人员校准数据中的陌生人误识率上限，不是模型训练准确率。括号内是将该
固定阈值用于测试人员后得到的陌生人实际误识率。

| 方案 | Rank-1均值（95%置信区间） | 严格档：库内人员正确识别并接受率（测试陌生人实际误识率） | 平衡档：正确识别并接受率（实际误识率） | 宽松档：正确识别并接受率（实际误识率） |
|---|---:|---:|---:|---:|
| FR0 ArcFace | **31.0%**（23.8%–38.1%） | **2.7%**（0.4%） | 4.5%（1.5%） | **7.5%**（5.7%） |
| FR1 AdaFace | 29.6%（22.1%–37.1%） | 1.4%（0.7%） | **5.1%**（2.8%） | 6.8%（12.0%） |
| FR2 ArcFace+GFPGAN | **31.0%**（23.8%–38.1%） | **2.7%**（0.4%） | 4.5%（1.5%） | **7.5%**（5.7%） |
| FR3 AdaFace+GFPGAN | 29.6%（22.1%–37.1%） | 1.4%（0.7%） | **5.1%**（3.5%） | 6.8%（12.0%） |

解释：

- ArcFace与AdaFace的Rank-1置信区间高度重叠，不能证明闭集能力存在显著差异；
- 在严格档下，ArcFace的库内人员正确识别并接受率更高；
- 在平衡档下，AdaFace略高，但两者置信区间仍重叠；
- AdaFace的宽松档阈值在测试集上产生约12%的陌生人实际误识率，说明训练人员到测试人员存在
  分数分布偏移；
- GFPGAN实际增强3个独立测试帧；这些帧在5次划分中形成15次重复评测记录，没有救回或破坏
  任何Rank-1结果。FR3在平衡档下的陌生人实际误识率为3.5%，略高于FR1的2.8%，但样本过少，
  仍不足以证明总体增益或风险。

本轮准确率实验因容器GPU在长跑前不可用而回退CPU；模型、输入和阈值逻辑不变，因此准确率结果
仍可使用，但本轮耗时不能作为GPU性能指标。

### 4.4 协议 C：MEVID 产品身份状态机回放

协议 C 用于贴近真实产品的身份层逻辑，不再提前按真值为每个身份建立均值模板。

#### 运行原则

1. `Canonical Subject Registry`、人脸库、人形库和步态库全部从空库开始；
2. 将官方 gallery tracklet 按固定顺序作为“建档视频流”逐条输入；
3. MEVID `person_id` 只保存在评测器中，运行时算法不可读取；
4. 第一条稳定且质量合格的未知轨迹创建新的 canonical `subject_id`；
5. 同一条轨迹中产生的人脸、人形和步态向量绑定到该 canonical subject：

```text
canonical_subject_id=42
  ├─ face vector rows
  ├─ body vector rows
  └─ gait vector rows
```

6. 后续轨迹分别查询三套向量索引，再将各本地命中映射回 canonical subject；
7. 只有达到各模态独立阈值的路线才能参与融合；
8. 多路高置信命中同一 canonical subject 时复用并补充模板；
9. 多路高置信冲突时进入 `grey/conflict`，禁止自动合并和写库；
10. 所有路线均未可靠命中且轨迹质量合格时，创建新的 canonical subject。

#### 与产品对齐的状态

协议 C 直接执行或严格复用：

- `identify_or_enroll` 的 `new/grey/hit`；
- multi-shot 模板补充与上限；
- 质量门控和低质量禁止写库；
- 各模态独立阈值；
- canonical subject 映射；
- 冲突不写库；
- query 阶段只查不建。

#### 指标

除 TPIR/FMR 外，增加：

- **建档覆盖率**：真实流式门控后有多少身份建立 canonical subject；
- **身份碎片率**：同一个真实身份被建立为多个 subject 的比例；
- **错误合并率**：不同真实身份被合并到同一 subject 的比例；
- **跨模态绑定准确率**：绑定到同一 subject 的 face/body/gait 是否属于同一真实身份；
- **首次建档延迟**：从身份首次出现到成功建档所需轨迹数；
- **模板污染率**：错误身份向量被写入已有 subject 的比例；
- **拒识率与灰区比例**。

### 4.5 与真实产品场景的接近程度

| 实验 | 与产品接近程度 | 能证明什么 |
|---|---|---|
| B2 固定锚点 | 低 | 模型本身在相同输入下谁更强 |
| B3 独立训练身份校准 | 中（模型评测） | 固定阈值能否从train泛化到test |
| C 状态机回放 | 中高（身份层） | 空库建档、主体映射、冲突和多模态身份逻辑 |
| 原始全帧端到端 | 最高 | 检测、跟踪、身份、融合和事件报告的完整产品效果 |

协议 C 比当前 A/B 更贴近产品，因为它从空库开始、隐藏真值、顺序建档并执行统一 subject 状态机；
但输入仍是已裁人物轨迹，不能代替原始全帧产品实验。最终仍需使用 MEVID/MEVA 原始视频或客户
授权监控视频，完整运行 YOLO、ByteTrack、人脸关联、身份建档与事件报告。

## 5. 解释边界

- 协议 A 回答“严格门控下，多模态能否在不显著增加误识的情况下救回身份”。
- 协议 B 回答“在相同、无 query 泄漏的最佳 gallery 锚点下，哪个人脸模型更能识别低质量监控脸”。
- 协议 B2 是更严谨的模型比较，但仍不是产品自动建档实验。
- 协议 C 回答“产品身份状态机从空库开始能否正确建档、复用和绑定三种模态”。
- 协议 A 的人脸覆盖率低是产品行为结果，不等于 ArcFace 识别能力差。
- 协议 B 的结果不能直接代表严格 `clear-only` 自动建档能力。
- 只有完成 B2、C 和原始全帧实验，才能分别回答“模型谁更强”“身份状态机是否可靠”和
  “完整产品是否有效”。
